# Irno Platform — Deployment Guide

This document covers production deployment of Irno Platform on a single VPS using **host Nginx** as the reverse proxy and Docker Compose for application containers.

---

## 1. Production Architecture

```
VPS (Ubuntu 22.04 LTS, 2 vCPU / 4 GB RAM / 2 GB swap)
IP: 185.213.167.39

  Host Nginx (:80, :443) — manages TLS certificates, proxies to Docker containers
  │
  ├── platform.irnocollege.com     → 127.0.0.1:13000 (hub-web)
  ├── platform.irnocollege.com/api → 127.0.0.1:14000 (hub-api)
  ├── cv.irnocollege.com           → 127.0.0.1:13002 (career-web)
  ├── cv.irnocollege.com/api       → 127.0.0.1:14000 (hub-api)
  ├── meet.irnocollege.com         → 127.0.0.1:13001 (meetino-web)
  ├── meet.irnocollege.com/api     → 127.0.0.1:14001 (meetino-api)
  ├── meet.irnocollege.com/socket.io → 127.0.0.1:14001 (meetino-api WebSocket)
  └── livekit.irnocollege.com      → 127.0.0.1:7880  (LiveKit signaling)

  Docker containers — all bind to 127.0.0.1 (NOT accessible directly from internet):
  ├── hub-api       127.0.0.1:14000:4000   (NestJS, Playwright/Chromium)
  ├── hub-web       127.0.0.1:13000:3000   (Next.js standalone)
  ├── career-web    127.0.0.1:13002:3002   (Next.js standalone)
  ├── meetino-api   127.0.0.1:14001:4001   (NestJS, Socket.IO)
  ├── meetino-web   127.0.0.1:13001:3000   (Next.js, container port is 3000)
  ├── livekit       127.0.0.1:7880:7880    (API/signaling — proxied via Nginx)
  │                 0.0.0.0:7881:7881/tcp  (ICE TCP — public, bypasses Nginx)
  │                 0.0.0.0:7882:7882/udp  (ICE UDP — public, bypasses Nginx)
  ├── postgres      internal only           (no host port)
  └── redis         internal only           (no host port)

  Persistent volumes:
  ├── postgres_data      → /var/lib/postgresql/data
  ├── redis_data         → /data
  └── hub_pdf_exports    → /app/storage/exports  (hub-api PDF files)

  Secrets (outside git, on VPS only):
  └── /opt/irno/secrets/livekit.prod.yaml   (chmod 0600 — LiveKit API key + secret)
```

**Production domains:**
- `https://platform.irnocollege.com` — Hub Web (admin + student portal + Irno ID)
- `https://cv.irnocollege.com` — Career Web (Irno CV / Career Studio)
- `https://meet.irnocollege.com` — Meetino (online classroom)
- `wss://livekit.irnocollege.com` — LiveKit WebSocket signaling (no browser UI)

**Architecture notes:**
- Host Nginx owns ports 80/443. Docker Compose does NOT run an Nginx container.
- All application container ports are bound to `127.0.0.1` — not accessible from the internet directly.
- LiveKit ICE ports 7881/tcp and 7882/udp are publicly exposed (WebRTC requires this). They bypass Nginx entirely.
- The LiveKit config file (`livekit.prod.yaml`) is generated by `first-deploy.sh` from `.env.prod` values and stored at `/opt/irno/secrets/livekit.prod.yaml`. It is never committed to git.

## 2. Prerequisites

**VPS:**
- Ubuntu 22.04 LTS (or Debian 12)
- Nginx installed on host (`sudo apt install nginx`)
- Docker Engine (not Docker Desktop)
- Docker Compose plugin (`docker compose`, not standalone `docker-compose`)
- Certbot installed on host (`sudo apt install certbot python3-certbot-nginx`)
- Git

**Network / Firewall (ufw or cloud security group):**
- Port 80 (TCP) — open inbound (HTTP, required for Certbot ACME challenge)
- Port 443 (TCP) — open inbound (HTTPS)
- Port 7881 (TCP) — open inbound (LiveKit ICE TCP — **required for WebRTC**)
- Port 7882 (UDP) — open inbound (LiveKit ICE UDP — **required for WebRTC**)
- All other ports: closed outbound from internet

**DNS (must be live before running Certbot):**
- A record: `platform.irnocollege.com` → VPS IP
- A record: `cv.irnocollege.com` → VPS IP
- A record: `meet.irnocollege.com` → VPS IP
- A record: `livekit.irnocollege.com` → VPS IP

## 3. Host Nginx Configuration

Nginx site configs for all four domains are in `infra/host-nginx/sites/`. Install them on the VPS:

```bash
# Copy site configs
sudo cp infra/host-nginx/sites/*.conf /etc/nginx/sites-available/

# Copy shared proxy params
sudo cp infra/host-nginx/proxy_params.conf /etc/nginx/proxy_params.conf

# Enable each site
for conf in /etc/nginx/sites-available/*.irnocollege.com.conf; do
    name=$(basename "$conf")
    sudo ln -sf "/etc/nginx/sites-available/$name" "/etc/nginx/sites-enabled/$name"
done

# Test and reload
sudo nginx -t && sudo nginx -s reload
```

After initial deployment and TLS certificate issuance (Section 4, step 5), Certbot will automatically upgrade these configs to HTTPS.

## 4. Environment Files

**⚠️ Security rule: Never commit real `.env` files.** The `.gitignore` excludes them. If a `.env` file appears in `git status`, run `git rm --cached <file>` before committing.

### Production compose env (primary)

The production Docker Compose stack reads **one file** for all shared secrets:

```
infra/docker/.env.prod          ← real production file, never commit
infra/docker/.env.prod.example  ← template, safe to commit
```

```bash
# Create on VPS — fill every GENERATE_WITH__ value before running first-deploy.sh
cp infra/docker/.env.prod.example infra/docker/.env.prod
nano infra/docker/.env.prod
```

### Per-app env templates (reference)

| App | Template |
|-----|----------|
| hub-api | `apps/hub-api/.env.production.example` |
| hub-web | `apps/hub-web/.env.production.example` |
| career-web | `apps/career-web/.env.production.example` |
| meetino-api | `apps/meetino-api/.env.production.example` |

In the Docker Compose production setup, per-app env vars are injected by `docker-compose.prod.yml` from `infra/docker/.env.prod`. The per-app template files are reference only.

**Secret generation:**
```bash
openssl rand -hex 32
```
Use for: `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`, `MEETINO_CLIENT_SECRET`, DB passwords, Redis password, and `SUPER_ADMIN_PASSWORD`.

**⚠️ Critical:** The correct hub-api CORS key is `API_CORS_ORIGINS` (plural, comma-separated list of all three browser origins). The singular form `API_CORS_ORIGIN` is silently ignored.

## 5. First Deployment

Three deployment scripts are in `infra/scripts/`:

| Script | Purpose |
|--------|---------|
| `migrate.sh` | Apply pending DB migrations (hub-api + meetino-api). Safe to run multiple times. |
| `first-deploy.sh` | One-shot initial setup: validate secrets → generate LiveKit config → migrate → seed → start all services. **Run once only.** |
| `deploy.sh` | Routine redeployment: build images → migrate → rolling restart. **Never seeds.** |

### Steps for the first deployment

```bash
# 1. Clone the repository on the VPS
git clone https://github.com/your-org/irno-platform.git /opt/irno
cd /opt/irno

# 2. Create and fill the production compose env file
cp infra/docker/.env.prod.example infra/docker/.env.prod
# Edit infra/docker/.env.prod — fill in ALL GENERATE_WITH__ values.
# Generate secrets with:  openssl rand -hex 32

# 3. Install and enable host Nginx site configs (see Section 3 above)
sudo cp infra/host-nginx/sites/*.conf /etc/nginx/sites-available/
sudo cp infra/host-nginx/proxy_params.conf /etc/nginx/proxy_params.conf
for conf in /etc/nginx/sites-available/*.irnocollege.com.conf; do
    sudo ln -sf "$conf" "/etc/nginx/sites-enabled/$(basename $conf)"
done
sudo nginx -t && sudo nginx -s reload

# 4. Run first-deploy.sh (validates secrets, generates LiveKit config,
#    builds images, migrates DBs, seeds, starts all services)
./infra/scripts/first-deploy.sh

# 5. Issue TLS certificates with host Certbot (DNS must be live before this step)
sudo certbot --nginx \
  -d platform.irnocollege.com \
  -d cv.irnocollege.com \
  -d meet.irnocollege.com \
  -d livekit.irnocollege.com \
  --email admin@irnocollege.com --agree-tos --non-interactive

# Certbot automatically rewrites Nginx configs to add HTTPS server blocks
# and sets up automatic renewal. Verify:
sudo nginx -t && sudo nginx -s reload

# 6. Run health checks (see Section 10)
CHECK_MODE=live ./infra/scripts/health-check.sh
```

Automatic certificate renewal is handled by the Certbot systemd timer installed by Certbot itself. No crontab entry needed.

## 6. LiveKit Configuration (Security Critical)

The LiveKit API key and secret are **never committed to git**. `first-deploy.sh` generates the LiveKit config file from `.env.prod` values at deployment time:

```
/opt/irno/secrets/livekit.prod.yaml   ← generated at deploy time, chmod 0600
```

The `livekit.prod.yaml` in the repository (`infra/docker/livekit/livekit.prod.yaml`) contains only a notice and is not used at runtime. The real config is volume-mounted from `/opt/irno/secrets/livekit.prod.yaml` in `docker-compose.prod.yml`.

**Required .env.prod values for LiveKit:**
```
LIVEKIT_API_KEY=<short alphanumeric identifier, e.g. APIirno2026>
LIVEKIT_API_SECRET=<openssl rand -hex 32>
LIVEKIT_WS_URL=wss://livekit.irnocollege.com
```

**ICE port requirements:** Ports 7881/tcp and 7882/udp must be reachable from the public internet. These bypass Nginx — they are published directly from the Docker container to the host. If these ports are blocked by the VPS firewall, video/audio will silently fail.

## 7. Routine Deployment

```bash
cd /opt/irno

# 1. Pull latest code
git pull origin main

# 2. Build + migrate + restart in one step
./infra/scripts/deploy.sh

# 3. Run health checks
CHECK_MODE=live ./infra/scripts/health-check.sh
```

`deploy.sh` always runs `migrate.sh` before restarting containers. **Never restart app containers before migrations complete.**

If you need to run only migrations (without restarting services):
```bash
./infra/scripts/migrate.sh
```

## 8. Database Migrations

| Command | When to use |
|---------|-------------|
| `prisma migrate deploy` | Every production deployment — applies pending migrations only |
| `prisma generate` | Inside Dockerfile build stage — never at container runtime |
| `prisma migrate dev` | Local development only — generates new migration files |
| `prisma migrate reset` | **Never in production** — drops all tables and data |
| `prisma db push` | **Never in production** — bypasses migration history |
| `prisma db seed` | First deployment only — creates the SUPER_ADMIN account |

Migrations in this project follow an idempotent pattern (`IF NOT EXISTS`). Running `migrate deploy` on an already-migrated database is safe.

## 9. Seeding the First Admin

Before running `db:seed`, set these three environment variables in `infra/docker/.env.prod`:

```
SUPER_ADMIN_MOBILE=09xxxxxxxxx
SUPER_ADMIN_EMAIL=admin@irnocollege.com
SUPER_ADMIN_PASSWORD=<generated with openssl rand -hex 32>
```

**⚠️ The development default password (`IrnoAdmin@2026`) must never be used in production.** The seed script logs a warning if it detects the default password.

The seed is idempotent via `upsert` — re-running it will not duplicate data.

## 10. Playwright and PDF Export

hub-api is the only app that uses Playwright to generate PDFs.

**In Docker (production):** No extra steps needed. The `apps/hub-api/Dockerfile` installs Alpine system Chromium packages at build time and configures:
```
PLAYWRIGHT_SKIP_BROWSER_DOWNLOAD=1
PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH=/usr/bin/chromium-browser
```

**Persistent volume requirement:**
`EXPORT_STORAGE_PATH` in hub-api `.env` must point to a directory backed by the `hub_pdf_exports` Docker volume. If this volume is missing or not mounted, PDF files will be written to the container's ephemeral filesystem and lost on every restart.

**Post-deploy smoke test:**
1. Log in as an admin at `https://platform.irnocollege.com`
2. Open Career Studio and create/open a resume
3. Trigger a PDF export
4. Verify the export reaches status `GENERATED` and downloads successfully

## 11. Meetino SSO

Meetino SSO (Phase 9.1) allows Meetino users to authenticate via Irno ID without a separate Meetino password. SSO code is complete but requires both apps running with matching secrets.

**Safe default:** `IRNO_HUB_SSO_ENABLED=false` in both hub-api and meetino-api `.env`. Enable only after verifying both apps are running and the shared secret is configured on both sides.

**To enable SSO after initial deployment:**
1. Set `MEETINO_ENABLED=true` and `IRNO_HUB_SSO_ENABLED=true` in the compose env
2. Set the same random value for `MEETINO_CLIENT_SECRET` (hub-api) and `IRNO_HUB_CLIENT_SECRET` (meetino-api)
3. Set `MEETINO_ALLOWED_REDIRECT_URLS=https://meet.irnocollege.com/auth/irno/callback`
4. Rebuild and restart both containers

## 12. Backup and Recovery

```bash
# Manual backup
/opt/irno/infra/scripts/backup.sh

# Automated daily backup (add to crontab)
0 3 * * * /opt/irno/infra/scripts/backup.sh
```

The backup script covers:
- PostgreSQL `irno_hub` database
- PostgreSQL `meetino_db` database
- PDF export files from hub-api storage

Recovery:
```bash
/opt/irno/infra/scripts/restore.sh <backup-file>
```

Backups are retained for 30 days by default.

## 13. Pre-Server Validation

Run these checks **before you have a VPS** to verify the repository is deployment-ready. No running services required.

### Static smoke test (CI-safe)

```bash
# From repo root
CHECK_MODE=static ./infra/scripts/health-check.sh
```

Verifies:
- All required files exist (Dockerfiles, Compose, Prisma schemas, example env templates, CI workflow, Nginx site configs)
- No real `.env` secrets are tracked by git
- `API_CORS_ORIGINS` (plural) is used in example files
- Deployment scripts are executable
- Docker Compose file parses without errors (when `docker` is available)
- Prisma schema syntax is valid (when `npx` is available)

Exit 0 = ready to deploy. Exit 1 = fix the listed items first.

### CI (GitHub Actions)

The workflow at `.github/workflows/ci.yml` runs on every push and pull request to `main`. It:
- Installs dependencies with `pnpm install --frozen-lockfile`
- Builds all shared packages
- Generates Prisma clients (no DB needed)
- Typechecks all five apps (strict — any error fails the job)
- Builds all five apps

No deploy, no secrets, no migrations run in CI.

### Health Checks (post-deployment)

```bash
# Default: checks production domains
CHECK_MODE=live ./infra/scripts/health-check.sh

# Override URLs for staging
CHECK_MODE=live \
  HUB_URL=https://platform.irnocollege.com \
  CAREER_URL=https://cv.irnocollege.com \
  MEETINO_URL=https://meet.irnocollege.com \
  ./infra/scripts/health-check.sh
```

Direct curl checks:
```bash
curl -f https://platform.irnocollege.com/api/v1/health
curl -f -o /dev/null -w "%{http_code}" https://platform.irnocollege.com
curl -f -o /dev/null -w "%{http_code}" https://cv.irnocollege.com
curl -f -o /dev/null -w "%{http_code}" https://meet.irnocollege.com
```

**Manual verification checklist:**
- [ ] Hub login page loads at `https://platform.irnocollege.com/auth/login`

- [ ] Career Studio loads at `https://cv.irnocollege.com`

## 14. Common Problems and Fixes

**CORS errors in browser:**
Check `API_CORS_ORIGINS` (plural) in hub-api `.env`. Must include all three browser origins: `https://platform.irnocollege.com,https://cv.irnocollege.com,https://meet.irnocollege.com`. The singular form `API_CORS_ORIGIN` is silently ignored.

**Login cookie not saved after login:**
- Verify the app is served over HTTPS in production
- Check that hub-api sets `secure: true` on cookies — this is automatic when `NODE_ENV=production`
- Verify `sameSite: lax` is correct for your browser's cross-site behaviour

**PDF export stays in PENDING status or returns error:**
- Check that the `hub_pdf_exports` Docker volume is mounted at `/app/storage/exports` in hub-api
- Check hub-api logs: `docker logs irno-hub-api`
- On Alpine/Docker: confirm the `chromium` system package installed correctly at build time

**Redis connection fails:**
- `REDIS_HOST` must be the compose service name (`redis`), not `localhost`
- Verify `REDIS_PASSWORD` matches the Redis container's `--requirepass` value
- Run: `docker exec irno-redis redis-cli -a YOUR_PASSWORD ping` — should return `PONG`

**LiveKit video/audio not working:**
- Verify **UDP port 7882** and **TCP port 7881** are open in the VPS firewall — this is the most common LiveKit failure
- Run: `lsof -i udp:7882` on the VPS to confirm the port is bound
- Check that `LIVEKIT_URL` in meetino-api `.env` uses `wss://livekit.irnocollege.com` (not the Meetino web domain)
- Check that `/opt/irno/secrets/livekit.prod.yaml` exists and has `chmod 0600`

**Frontend shows wrong URLs (e.g. localhost instead of production domain):**
`NEXT_PUBLIC_*` variables are baked into the JavaScript bundle at `next build` time. If they were wrong or missing at build time, rebuilding the Docker image is the only fix.

**meetino-web container port is 3000, not 3001:**
The meetino-web container exposes port 3000 internally. The host binding is `127.0.0.1:13001:3000` — host port 13001 maps to container port 3000. Nginx proxies to `127.0.0.1:13001`. Do not change the container binding.

**Meetino SSO broken after Hub deployment:**
- Verify `MEETINO_CLIENT_SECRET` in hub-api `.env` matches `IRNO_HUB_CLIENT_SECRET` in meetino-api `.env`
- Verify the callback URL in `MEETINO_ALLOWED_REDIRECT_URLS` matches exactly (scheme, host, path — no trailing slash)
- Both apps must have `IRNO_HUB_SSO_ENABLED=true` (safe default during initial deploy is `false`)

**Host Nginx "502 Bad Gateway":**
- Verify the Docker container is running: `docker ps | grep irno`
- Verify the port binding: `ss -tlnp | grep 130` (should show 13000, 13001, 13002)
- Verify the upstream address in the Nginx site config matches the actual container binding

**TLS certificate renewal:**
Certbot systemd timer runs automatically (`systemctl status certbot.timer`). After renewal, Nginx reloads automatically. Verify with: `sudo certbot renew --dry-run`
